La tarea BE-001 Crear API Registro de Institución solicita desarrollar el backend completo para el registro de una institución deportiva. En términos prácticos, esto es lo que debe entregar:

1. Crear el endpoint REST
POST /api/instituciones

Debe recibir la información enviada desde el frontend.

2. Recibir la información de la institución

La API debe aceptar como mínimo:

Nombre de la institución.
Representante legal.
Carácter.
País.
Sede.
Página web (opcional).
Correo electrónico.
Teléfono.
Disciplinas deportivas.
Categorías de edad por cada disciplina.
3. Validar la información

Antes de guardar debe verificar:

El nombre es obligatorio.
El representante legal es obligatorio.
El carácter es obligatorio.
El país es obligatorio.
La sede es obligatoria.
El correo tiene formato válido.
El correo no está registrado.
El nombre de la institución no está registrado.
El teléfono tiene un formato válido.
La página web, si existe, tiene formato válido.
Debe existir al menos una disciplina.
Cada disciplina debe tener al menos una categoría de edad.
No permitir disciplinas repetidas.
No permitir categorías repetidas dentro de una disciplina.
4. Aplicar la lógica de negocio

La API debe:

Verificar si la institución ya existe.
Verificar si el correo ya existe.
Verificar que las disciplinas existan.
Crear la institución.
Asociar las disciplinas.
Asociar las categorías de edad correspondientes.
Guardar toda la información en la base de datos.
5. Persistencia

Implementar la persistencia usando Prisma y PostgreSQL para:

Institución.
Relaciones con disciplinas.
Relaciones con categorías de edad.
6. Manejo de errores

La API debe responder correctamente según el caso:

201 Created: Registro exitoso.
400 Bad Request: Datos inválidos o incompletos.
404 Not Found: Disciplina no encontrada.
409 Conflict: Nombre o correo ya registrados.
500 Internal Server Error: Error inesperado.
7. Crear los componentes del backend

Debe incluir:

Controller.
Service.
DTO de entrada.
DTO de respuesta.
Repositorio (si la arquitectura del proyecto lo utiliza).
Validaciones.
Manejo global de excepciones.
8. Documentación

El endpoint debe aparecer documentado en Swagger/OpenAPI con:

Descripción.
Parámetros.
Cuerpo de la petición.
Respuestas HTTP.
9. Auditoría

Registrar logs o eventos del proceso de registro de la institución.

10. Pruebas

Realizar:

Pruebas unitarias.
Pruebas de integración.
Entregables finales esperados

Al finalizar la tarea, el proyecto debe contar con:

✅ Endpoint POST /api/instituciones.
✅ DTO de entrada y salida.
✅ Validaciones de datos y reglas de negocio.
✅ Servicio con la lógica de registro.
✅ Persistencia en PostgreSQL mediante Prisma.
✅ Relaciones entre institución, disciplinas y categorías de edad.
✅ Manejo adecuado de excepciones y códigos HTTP.
✅ Documentación en Swagger/OpenAPI.
✅ Logs de auditoría.
✅ Pruebas unitarias e integración.
